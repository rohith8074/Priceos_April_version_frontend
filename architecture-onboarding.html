01 · The Problem

Why "Pre-Push" Sync Fails

Most PMS integrations (including the current PriceOS V1 model) pull all available data from Hostaway immediately on connection. This creates friction at every layer of the product.

🐌

Slow Onboarding

Syncing 100+ properties, 1-year calendars, and all reservations on Day 1 takes minutes. New users abandon the flow before it completes.

🔒

No Data Isolation

Every agent (Pricing, Guest, Benchmark) has access to all collections. A bug in the Guest Agent could accidentally affect pricing data.

💸

Wasted API Calls

Hostaway charges per API call. Fetching reservation history for un-activated properties burns quota that users don't benefit from.

😵

Cognitive Overload

New users land on a dashboard showing 200 proposals for properties they haven't reviewed yet. No context. No guided setup. They're lost.

02 · New Architecture

Lazy Tables + Agent Lenses

Data lives in one Central MongoDB store. It is only fetched when a specific Agent needs it. Each Agent has a read-only "Lens" — a strict collection allowlist. No agent can read outside its scope.

🔑 Key Principle

Agents do not "own" data. They query a read-only projection of the Central Layer. If the Guest Agent crashes, pricing data is completely unaffected. This is called a Data Mesh Pattern.

04 · Onboarding Flow

5-Step Wizard: Progressive Activation

New users go through a guided wizard. Data is fetched from Hostaway incrementally — only the minimum required at each step. Properties are activated one-by-one, not bulk-synced.

05 · Agent Isolation

Collection Allowlist per Agent

Each agent is initialized with a strict dataScope configuration. A middleware wrapper around MongoDB enforces this at query time — agents cannot read outside their scope even if called incorrectly.

Agent Can Read (Allowed) Can Write Cannot Access
🧠 PricingAgent listings · calendar · market_templates proposals (write new only) conversations · financials · reservations
💬 GuestAgent conversations · reservations · listings · calendar conversations (reply drafts) proposals · financials
🛡 GuardrailAgent listings (rules only) · proposals (pending) proposals (status update) conversations · calendar · reservations
📊 BenchmarkAgent listings · calendar · reservations insights (write new only) conversations · proposals · financials
💰 FinanceModule financials · reservations · listings financials conversations · proposals · calendar
✅ Implementation Note

The dataScope config is enforced inside a wrapper utility like lib/db/scoped-query.ts. When any agent calls scopedQuery(agentName, collection, filter), the wrapper checks the allowlist and throws a security error if the collection is not permitted — before the MongoDB query runs.

06 · Code Spec

Enforcement: Scoped Query Middleware

// lib/db/scoped-query.ts

const AGENT_SCOPES = {
  PricingAgent:    { read: ['listings', 'calendar', 'market_templates'], write: ['proposals'] },
  GuestAgent:      { read: ['conversations', 'reservations', 'listings', 'calendar'], write: ['conversations'] },
  GuardrailAgent:  { read: ['listings', 'proposals'],                   write: ['proposals'] },
  BenchmarkAgent:  { read: ['listings', 'calendar', 'reservations'],   write: ['insights'] },
  FinanceModule:   { read: ['financials', 'reservations', 'listings'],  write: ['financials'] },
};

export async function scopedRead(agent: keyof typeof AGENT_SCOPES, collection: string, filter = {}) {
  const allowed = AGENT_SCOPES[agent].read;
  if (!allowed.includes(collection)) {
    throw new Error(`[ScopedQuery] ${agent} is not allowed to read '${collection}'`);
  }
  return db.collection(collection).find(filter).toArray();
}

export async function scopedWrite(agent: keyof typeof AGENT_SCOPES, collection: string, doc: object) {
  const allowed = AGENT_SCOPES[agent].write;
  if (!allowed.includes(collection)) {
    throw new Error(`[ScopedQuery] ${agent} is not allowed to write to '${collection}'`);
  }
  return db.collection(collection).insertOne(doc);
}
07 · Data Model Addition

Onboarding State in Organization Model

We need to add an onboarding field to the Organization schema to track wizard progress and persist selected properties before activation.

// Addition to IOrganization interface
onboarding: {
  step: 'connect' | 'select' | 'market' | 'strategy' | 'complete';
  hostawayAccountId?: string;
  selectedListingIds: string[];   // Properties the user checked in Step 2
  activatedListingIds: string[];  // Properties where full sync completed
  completedAt?: Date;
};
💡 Why track onboarding steps?

If a user drops off at Step 3, the next login resumes the wizard from Step 3 — they don't have to re-enter their API key or re-select their properties. This is a major UX improvement over re-starting from scratch.
