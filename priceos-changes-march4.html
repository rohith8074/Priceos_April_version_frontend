1. Server-Side Guardrails (Defense in Depth)

Backend Code

Problem

Floor/ceiling clamping and ±50% swing checks only existed in prompts. If anyone changed the prompt or the LLM ignored instructions, proposals could violate business rules — prices below floor, above ceiling, or with dangerous swings.

Solution

Added an enforceGuardrails() function in TypeScript that runs after the LLM returns its response and before sending proposals to the frontend. This is the "Layer 3" defense — it cannot be bypassed by prompt changes.

📄 src/app/api/chat/route.ts → lines 497–555

What It Does (4 Checks)

Check Rule Action
HARD GATE Floor Clamp proposed_price < floor_price Set to floor_price. Log in server_notes
HARD GATE Ceiling Clamp proposed_price > ceiling_price Set to ceiling_price. Log in server_notes
REJECT Swing Limit |change_pct| > 50% Set guard_verdict = "REJECTED"
COMPUTE Risk Level <5% = low, 5-15% = medium, >15% = high Overwrite LLM's risk level with deterministic value

Data Flow

LLM Response
→
extractAgentMessage()
→
🛡️ enforceGuardrails()
→
Save to DB
→
Return to Frontend
+ const floorPrice = Number(propertyDataPayload?.property?.floor_price || 0);
+ const ceilingPrice = Number(propertyDataPayload?.property?.ceiling_price || 0);
+ let enforcedProposals = parsedJson?.proposals || null;
+
+ if (enforcedProposals && Array.isArray(enforcedProposals) && (floorPrice > 0 || ceilingPrice > 0)) {
+   enforcedProposals = enforceGuardrails(enforcedProposals, floorPrice, ceilingPrice);
+ }

2. Date Range Injection — analysis_window

Critical Fix

Problem

The user selected March 4 – March 18 in the UI, but the agent responded with analysis for March 1 – March 15. The date range was never sent to the LLM — it had to guess from the data it saw.

Root Cause

The propertyDataPayload injected into the LLM prompt had property, metrics, benchmark, market_events, and recent_reservations — but no date range field.

Solution

Added analysis_window as the first field in the payload:

📄 src/app/api/chat/route.ts → lines 238–242
Before 
propertyDataPayload = {
  property: { ... },
  metrics: { ... },
  ...
}
After 
propertyDataPayload = {
  analysis_window: {
    from: "2026-03-04",
    to: "2026-03-18"
  },
  property: { ... },
  metrics: { ... },
  ...
}

Prompt Update

Added to the CRO Router prompt with bold emphasis:

- `analysis_window`: `from` (YYYY-MM-DD), `to` (YYYY-MM-DD)
   — THIS IS THE Date Range the user selected.
   Use these dates as the start and end of your analysis window.
   NEVER use any other dates.

3. Chat History Scoped to Date Range

Backend + Frontend

Problem

Changing the date range from Mar 3–17 to Mar 4–18 still showed the old chat messages. The history API loaded messages by listingId only — it didn't check the date range.

Solution

The session ID already encodes the date range (e.g., property-225-2026-03-04-2026-03-18). We now pass this to the history API to scope the query.

📄 src/app/api/chat/history/route.ts
📄 src/components/chat/unified-chat-interface.tsx

Changes

File What Changed
history/route.ts Added sessionId query parameter. When provided, filters messages by both listingId AND sessionId
unified-chat-interface.tsx Frontend now passes sessionId to the history API. Also added dateRange to the useEffect dependencies so chat refreshes immediately on date change

Result

User selects Mar 4–18
→
Session ID: property-225-...-03-04-...-03-18
→
History API: WHERE sessionId = ?
→
Empty chat (clean slate)

6. PriceGuard Pricing Formula — Fallback & Differentiation

Prompt

Problem

All 8 proposals came back as AED 600 (the floor price). Every date got the same price — no differentiation between weekdays, weekends, and event dates. The formula's base rate was 0 when benchmark data was missing.

Solution — Two Changes

📄 updated_prompts_2/05-price-guard.md

A. Fallback when benchmark rates are 0:

FALLBACK: If the benchmark rate is 0 or missing:
  - Use MAX(property.current_price, property.floor_price) as base
  - For weekends: base × 1.10
  - For event dates: base × 1.20

B. Differentiation Rule:

**CRITICAL: Differentiation Rule**
- Event dates MUST be priced HIGHER than non-event dates
- Weekend rates MUST be different from weekday rates
- If clamping makes everything equal to floor, increase
  event dates to floor × event_factor

Expected Pricing After Fix

Date Day Type Before (All Same) After (Differentiated)
Mar 4 (Tue) Weekday AED 600 AED 600
Mar 5 (Wed) Weekday AED 600 AED 600
Mar 6 (Thu, Art Dubai) Event AED 600 AED 690+
Mar 7 (Fri, Art Dubai) Weekend + Event AED 600 AED 759+
Mar 8 (Sat, Art Dubai) Weekend + Event AED 600 AED 759+

7. Guardrails Tooltip & Chat Disable

UI

Problem

When guardrails (floor/ceiling) are not set for a property, the user could still see old chat messages. The native HTML title tooltip was invisible on disabled buttons and had a 1-second delay.

Solution

Replaced native title with Radix <Tooltip> component showing an amber popup on hover. Disabled the chat input when guardrails aren't set.

8. Max Tokens: 2000 → 4000

Prompt

Problem

The CRO Router's max_tokens was set to 2000. A full analysis with 10 sections + 8 proposals in structured JSON easily exceeds 2000 tokens, causing the response to be truncated — which is why sections like booking velocity and competitor data were missing.

Solution

Increased to 4000 tokens to accommodate the comprehensive response format.
