01 · The Problem We're Solving

What PriceOS Is Missing Today

The current pricing engine in pipeline.ts and waterfall.ts is rule-based with internal occupancy data. It completely lacks external market intelligence — meaning it has no idea what competitors are charging, what the market demand is, or what's happening in Dubai that could justify repricing tonight.

CRITICAL GAP

No Market Benchmarks

The pricing engine uses the host's own historical average as the base price. It has zero knowledge that comparable 2BR properties in Dubai Marina are averaging AED 850/night this month.

CRITICAL GAP

No Forward Demand Signal

The engine cannot see that next Saturday is at 87% market occupancy (high pacing) vs the normal 52% — which should trigger an automatic rate surge. This data simply doesn't exist in PriceOS today.

MAJOR GAP

No Comp Set Feature

PriceOS has no way to identify, track, or benchmark against competing Airbnb listings. Hosts have no visibility into how they are priced relative to their actual competition.

MAJOR GAP

Perplexity Gives Text, Not Numbers

The existing InternetResearchAgent uses Perplexity to search "What are current rates in Dubai Marina?" — and gets a paragraph of text back. Airbtics gives actual structured numbers: p50 ADR = AED 842.

03 · Step-by-Step Flow

How the Pricing Engine Gets Market Data

Here's the exact sequence of events from the moment the pricing engine is triggered for a Dubai property to when the final proposal is stored in MongoDB.

🏠

Step 1 — Engine Triggered for a Listing

The pipeline runs for a listing (e.g. a 2-bedroom in Dubai Marina). The first thing it does is resolve the market context — today this is empty; after integration, it fetches from the cache or calls Airbtics.

  • Reads listing config (bedrooms, orgId, priceFloor, etc.) from MongoDB
  • Determines the Dubai market ID (static, stored once in MarketTemplate)
  • Checks the cache for existing market data before any API call
📊

Step 2 — Fetch Airbtics Quantitative Data (with cache)

Airbtics is called only if cache is stale. Three endpoints are called in parallel for Dubai 2BR:

  • Market Summary → real LTM ADR, occupancy %, active listing count
  • Monthly Metrics → 12-month history of ADR percentiles (p25/p50/p75)
  • Future Pacing → forward-looking occupancy and ADR by check-in date
Cache hit → 0ms, 0 credits  Cache miss → ~300ms, 3 API calls
🔍

Step 3 — Fetch Perplexity Qualitative Data (events only)

Perplexity is called for things Airbtics cannot tell you — what's happening in the real world. After integration, Perplexity is restricted to event/news research only. Market rate queries are removed and replaced by Airbtics.

  • Searches for upcoming Dubai events in the next 60 days
  • Identifies conferences, festivals, public holidays, Ramadan dates
  • Returns structured findings: event name, dates, expected impact, confidence
🔀

Step 4 — Synthesize: Build the MarketContext Object

Both data sources are merged into a single MarketContext TypeScript object. This is the "source of truth" that the pricing waterfall will consume for each day in the 365-day loop.

interface MarketContext {
  marketId: string;            // e.g. "2286" (Dubai from Airbtics)
  bedrooms: number;           // 2 (from Listing config)

  // 📊 FROM AIRBTICS
  marketOccupancyPct: number; // LTM market occupancy (e.g. 71%)
  p50ADR: number;             // Median daily rate (e.g. AED 842)
  p75ADR: number;             // 75th percentile (e.g. AED 1,240)
  futurePacing: PacingEntry[];  // Per-date forward demand data
  monthlyTrend: MonthlyMetric[]; // 12-month seasonal ADR history

  // 🔍 FROM PERPLEXITY
  upcomingEvents: EventFinding[]; // Confirmed Dubai events
  demandSignal: "high" | "medium" | "low";
  researchSummary: string;    // Human-readable context for proposal reasoning
}
⚙️

Step 5 — Waterfall Consumes MarketContext Per Day

The 365-day pricing loop in pipeline.ts now has access to full market data per day. For each date, it checks the pacing data and any active events to apply market-driven adjustments on top of the existing rules.

  • If pacing occupancy > 80% for a date → trigger rate increase
  • If a high-impact event overlaps the date → layer on top of base event rule
  • The proposal reasoning now reads: "Market pacing at 87% (GITEX conference), +15% applied"
💾

Step 6 — Save Proposals with Full Reasoning

The enriched proposals are written to InventoryMaster in MongoDB as before, but now the reasoning field contains specific, data-backed explanations — not just "high occupancy" but "Market ADR p50 = AED 842, your current rate is AED 700 (17% below market). Pacing for this date is 83%. Proposed rate: AED 950."

04 · API Endpoints

Exactly Which Airbtics Endpoints We Use

Out of the 17 available Airbtics endpoints, PriceOS will use 6. Below is a detailed breakdown of each one — what it does, what parameters to send, what data we get back, and how it's used in the code.

🔧 One-Time Setup (Run Once, Store in DB)

Before anything, we need Dubai's market_id. Call GET /markets/search?query=Dubai&country_code=AE once, get the ID (e.g. 2286), and store it in the MarketTemplate MongoDB document as airbticsMarketId. Every subsequent call uses this ID — no repeat lookups needed.

GET /markets/summary?market_id=2286&bedrooms=2 🔴 Critical 💳 Low Cost
Market Summary — The Baseline Anchor

Gets the Last-Twelve-Month (LTM) aggregate KPIs for the Dubai 2-bedroom market. This is the single most important call — it tells us what the market is actually doing at a high level.

Response data we use
  • average_daily_rate — market median ADR (AED)
  • occupancy — market LTM occupancy % (e.g. 0.71)
  • revenue — median monthly revenue
  • active_listings_count — supply pressure indicator
Where it's used in PriceOS
  • Sets p50ADR in MarketContext
  • Populates Market Insights dashboard card
  • Prices proposals relative to market median
  • Tells hosts if they're above/below market
✅ Cache for 24 hours

LTM market data changes very slowly. One call per market+bedroom combination per day is more than sufficient.

GET /markets/metrics/all?market_id=2286&bedrooms=2&number_of_months=12 🔴 Critical 💳 Medium Cost
Monthly Metrics — Seasonality Intelligence

Returns a month-by-month array of market performance metrics for the past 12 months. This is how we understand Dubai's seasonal pricing rhythm — when are peak months, shoulder months, and low season?

Response data we use
  • month — YYYY-MM period
  • p25_adr, p50_adr, p75_adr — ADR percentiles
  • occupancy — monthly occupancy rate
  • revenue_yoy_change_pct — year-over-year trend
Where it's used in PriceOS
  • Seasonal adjustment multiplier per month
  • Dashboard: 12-month market trend chart
  • Validates if a pricing rule makes sense seasonally
  • Feeds the "Market Trends" section in Insights
💡 How This Improves Pricing

Without this data, the engine treats all months equally. With it, we know Dubai December–February has a p50 ADR of AED 1,100 vs AED 580 in July — a nearly 2× seasonal difference that the current engine ignores.

GET /markets/metrics/future-pacing?market_id=2286&bedrooms=2 🔴 Critical 💳 Medium Cost
Future Pacing — The Most Powerful Endpoint

Returns forward-looking occupancy and ADR for each upcoming check-in date in the market. This is the gold standard for demand forecasting — instead of guessing, you see exactly how booked-up the Dubai market is for any specific future date, right now.

Response data we use
  • date — specific check-in date (YYYY-MM-DD)
  • occupancy — current booking rate for that date
  • adr — average rate competitors are charging
  • vs_ly_occupancy — vs same date last year
Where it's used in PriceOS
  • Per-date demand signal in the 365-day pricing loop
  • Calendar heatmap (red=hot, blue=cold demand)
  • Triggers surge pricing when market occupancy > 75%
  • Triggers discount when pacing is unusually low
⚡ Real Example: Impact on a Proposal

Pacing data shows Dec 31st Dubai at 94% occupancy. Without pacing, engine prices at AED 850. With pacing, engine sees 94% market-wide demand and surges to AED 1,350 — correctly aligned to what the market will bear that night.

POST /listings/search/bounds 🟠 High Value 💳 Medium Cost
Listing Search by Bounds — Comp Set Builder

POST a lat/long bounding box around a property (e.g. ±0.05° from Dubai Marina coordinates) and get back a list of all nearby Airbnb listings with their performance stats. This enables a real Comp Set feature.

Request body we send
  • bounds.ne_lat, bounds.ne_lng
  • bounds.sw_lat, bounds.sw_lng
  • bedrooms — filter by bedroom count
  • page — paginate results
Response data we use
  • listings[].listing_id — Airbnb listing ID
  • listings[].ltm_occupancy — competitor occupancy
  • listings[].ltm_adr — competitor ADR
  • listings[].ltm_revenue — competitor revenue
📌 New Feature Unlocked: Comp Set

This call enables a "Your Competition" panel in the dashboard — showing the host their top 5–10 nearest similar properties with their actual LTM performance. Cache for 24 hours per property (comp sets don't change overnight).

POST /report/all 🔵 Onboarding 💳 Higher Cost
Revenue Report — Onboarding Projection Engine

Pass a property's coordinates + bedroom/bathroom counts and get back a full revenue projection with comparable listings. Call this once during onboarding — then fetch the report by ID.

Request body we send
  • latitude, longitude
  • bedrooms, bathrooms
  • accommodates — max guests
Response data we use
  • report_id → then GET /report?report_id=...
  • Estimated annual revenue range
  • Comp listing data (10–20 comparables)
  • Seasonal revenue breakdown
✅ When to Call This

Only during onboarding Step 4 ("Strategy") for each newly activated property. Show the host: "Based on your location, comparable properties earn AED 85,000–120,000/year." Store the result permanently in the Listing document — never re-call unless the property data changes.

GET /listings/metrics/all?listing_id=AIRBNB_ID 🟢 Optional 💳 Low Cost
Single Listing Deep Dive — Competitor Profiling

Get complete 12-month analytics for one specific Airbnb listing ID. Used when a host wants to monitor a specific competitor or when seeding the comp set with detailed profiles.

Response data we use
  • Full 12-month occupancy and ADR history
  • Seasonal pricing patterns
  • Review score and count
  • Listing amenity tier inference
Where it's used in PriceOS
  • Comp card in the Benchmark Agent view
  • Enriches comp set with seasonal context
  • Called on demand (user clicks a competitor)
05 · Cost & Credits

API Pricing & Cost Estimation for Dubai

Airbtics uses a pay-as-you-go credit model. You top up credits and endpoints consume credits per call. The exact credit amount per endpoint is visible in your Airbtics dashboard after registration. The data refreshes weekly (every Thursday). Here's a realistic cost model for PriceOS using Dubai as the primary market.

⚠️ Important: Credits Are Consumed Per API Call

This means caching is not optional — it is a core cost control mechanism. Without caching, every pricing run for every property would consume credits. With smart caching shared across all listings in the same market+bedroom bucket, 50 property runs can share 1 set of API responses. This drops costs by ~98%.

Endpoint Cost Tier Cache TTL Calls/Day (Dubai, all bedrooms) Why We Cache Aggressively
/markets/search 🆓 Near-Free Forever (static) 1 call total (one-time setup) Dubai's market_id never changes
/markets/summary 💳 Low 24 hours 6 calls/day (6 bedroom types: 0,1,2,3,4,5+) LTM data updates weekly; daily cache is more than fresh enough
/markets/metrics/all 💳 Medium 24 hours 6 calls/day (one per bedroom count) Monthly data static within a week; cache 24h safely
/markets/metrics/future-pacing 💳 Medium 6 hours 24 calls/day (6 bedroom × 4 refreshes) Pacing changes as bookings come in; refresh every 6 hours for accuracy
/listings/search/bounds 💳 Medium 24 hours ~20 calls/day (per activated property) Comp sets are stable; no need to refresh more than once per day
/report/all 💳 Higher Permanent (store in DB) 1 per newly onboarded property One-time calculation at onboarding; result stored forever in Listing doc
/listings/metrics/all 💳 Low 24 hours On-demand (user action) Only called when user explores a specific competitor
💰 Realistic Monthly Cost Estimate

With aggressive caching: the steady-state daily volume is under 60 API calls per day shared across the entire Dubai market (all listings with the same bedroom type share the same cached response). At standard Airbtics pay-as-you-go rates, this is roughly $5–$30/month depending on credit tiers — a negligible cost that unlocks enterprise-grade market intelligence for all users.

06 · Data Value Ranking

What Actually Matters for Dubai Pricing

Not all data is equally valuable. Here's a ranked breakdown of what each data type does for PriceOS and why Dubai specifically benefits from it.

HIGHEST VALUE

Future Pacing Data

What it is: Per-date forward occupancy and ADR for the Dubai market.

Why it's #1: Dubai is an event-driven market — GITEX, Art Dubai, Formula E, New Year's Eve, Eid holidays — demand swings are enormous and predictable from pacing data weeks in advance.

Impact: Enables the engine to surge prices 2–6 weeks ahead of peak dates before the host even knows there's an event. This is where the most revenue is captured.

VERY HIGH VALUE

Market ADR Percentiles (p50 / p75)

What it is: Monthly ADR at the 25th, 50th, 75th, and 90th percentile for Dubai by bedroom count.

Why it matters: Many Dubai hosts are chronic underpricers. Knowing that their 2BR in JBR should command AED 900/night (p50) when they're charging AED 620 is an immediate, measurable improvement.

Impact: This single data point often justifies a 15–30% rate increase for underpriced hosts — the core value proposition of PriceOS.

HIGH VALUE

Dubai Events (Perplexity)

What it is: Structured findings from Perplexity about upcoming events in Dubai: conferences, festivals, public holidays.

Why it matters: Pacing data shows demand is high, but Perplexity explains why. This context makes the proposal reasoning meaningful to hosts.

Impact: Host sees: "GITEX Technology Week (Oct 14–18, 87,000 attendees) — +22% rate recommended." This is trusted, actionable, and explainable.

MEDIUM VALUE

Comp Set (Nearby Listing Data)

What it is: LTM performance of the 10 nearest comparable Airbnb listings, filtered by bedroom count.

Why it matters: Benchmark Agent gains real competitor data instead of market-wide averages — critical for premium properties that sit above the p50 price point.

Impact: Unlocks the full Benchmark Agent functionality. Hosts can see that their penthouse in Downtown Dubai competes against 7 specific listings, not just the area average.

2 src/lib/airbtics/client.ts New File

Core API wrapper. All Airbtics HTTP calls go through this file. Handles auth, error handling, and typed responses.

const BASE = 'https://crap0y5bx5.execute-api.us-east-2.amazonaws.com/prod';
const KEY  = process.env.AIRBTICS_API_KEY!;

async function get<T>(path: string, params: Record<string, string> = {}): Promise<T> {
  const url = new URL(`${BASE}${path}`);
  Object.entries(params).forEach(([ k, v ]) => url.searchParams.set(k, v));
  const res = await fetch(url.toString(), {
    headers: { 'x-api-key': KEY },
    next: { revalidate: 0 } // we handle caching ourselves
  });
  if (!res.ok) throw new Error(`Airbtics ${res.status}: ${path}`);
  return res.json();
}

export const airbtics = {
  // One-time setup
  searchMarket: (query: string, countryCode: string) =>
    get('/markets/search', { query, country_code: countryCode }),

  // Core market intelligence — used in pipeline
  getMarketSummary: (marketId: string, bedrooms: string) =>
    get('/markets/summary', { market_id: marketId, bedrooms }),

  getMarketMetrics: (marketId: string, bedrooms: string, months = '12') =>
    get('/markets/metrics/all', { market_id: marketId, bedrooms, number_of_months: months }),

  getFuturePacing: (marketId: string, bedrooms: string) =>
    get('/markets/metrics/future-pacing', { market_id: marketId, bedrooms }),

  // Comp set builder
  searchListingsByBounds: (bounds: { ne_lat: number; ne_lng: number; sw_lat: number; sw_lng: number }, bedrooms: number) =>
    fetch(`${BASE}/listings/search/bounds`, {
      method: 'POST',
      headers: { 'x-api-key': KEY, 'Content-Type': 'application/json' },

    fetch(`${BASE}/report/all`, {
      method: 'POST',
      headers: { 'x-api-key': KEY, 'Content-Type': 'application/json' },
      body: JSON.stringify({ latitude: lat, longitude: lng, bedrooms })
    }).then(r => r.json()),

  getReport: (reportId: string) =>
    get('/report', { report_id: reportId }),
};
3 src/lib/airbtics/market-context.ts New File

Builds the unified MarketContext object used by the pricing pipeline. Contains all caching logic.

import { airbtics } from './client';

// In-memory cache (can be upgraded to Redis later)
const cache = new Map<string, { data: any; expiresAt: number }>();

async function cached<T>(key: string, ttlMs: number, fn: () => Promise<T>): Promise<T> {
  const hit = cache.get(key);
  if (hit && hit.expiresAt > Date.now()) return hit.data;
  const data = await fn();
  cache.set(key, { data, expiresAt: Date.now() + ttlMs });
  return data;
}

const H2  = 2  * 60 * 60 * 1000;  // 2 hours
const H6  = 6  * 60 * 60 * 1000;  // 6 hours → pacing
const H24 = 24 * 60 * 60 * 1000; // 24 hours → metrics + summary

export async function buildMarketContext(marketId: string, bedrooms: number) {
  const br = String(bedrooms);
  const key = `${marketId}:${br}`;

  const [ summary, metrics, pacing ] = await Promise.all([
    cached(`summary:${key}`, H24, () => airbtics.getMarketSummary(marketId, br)),
    cached(`metrics:${key}`, H24, () => airbtics.getMarketMetrics(marketId, br, '12')),
    cached(`pacing:${key}`,  H6,  () => airbtics.getFuturePacing(marketId, br)),
  ]);

  // Build a fast lookup map: date → { occupancy, adr }
  const pacingMap = new Map<string, { occupancy: number; adr: number }>();
  for (const entry of pacing?.data ?? []) {
    pacingMap.set(entry.date, { occupancy: entry.occupancy, adr: entry.adr });
  }

  return {
    marketId,
    bedrooms,
    p50ADR:              summary?.average_daily_rate ?? 0,
    marketOccupancyPct: (summary?.occupancy ?? 0) * 100,
    monthlyMetrics:     metrics?.data ?? [],
    pacingMap,           // O(1) lookups in the pipeline loop
  };
}
