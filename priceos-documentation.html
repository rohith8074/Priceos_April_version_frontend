01 System Architecture Overview

How data flows from Hostaway PMS through AI agents to the user interface.

┌─────────────────────────────────────────────────────────────────────┐ │ PriceOS Architecture │ └─────────────────────────────────────────────────────────────────────┘ Hostaway PMS API ──GET──▶ Python Sync Backend ──INSERT──▶ PostgreSQL (Neon) (Listings, (FastAPI @ :8000) (10 tables) Calendar, Syncs listings, Reservations) calendar, reservations PostgreSQL ──SELECT──▶ Next.js API Routes ──JSON Payload──▶ Lyzr Agent Studio (/api/chat) (CRO Router Agent) (/api/market-setup) (Sub-Agents) (/api/events) (/api/benchmark) Lyzr Agent Studio ──Structured JSON──▶ Next.js API ──Response──▶ React Frontend CRO Router (/api/chat) Chat UI → PropertyAnalyst Parses JSON Sidebar → BookingIntelligence Saves to DB Proposals → MarketResearch Returns to UI Calendar → PriceGuard
💡 Key Design Decision: Data Injection on First Message On the user's first message in a chat session, route.ts fetches ALL property data from PostgreSQL and injects it directly into the Lyzr prompt. The agent remembers this data for the entire session. Subsequent messages do NOT re-fetch data.

03 Agent Orchestration — All 9 Agents

Each agent has a specific role, model, and structured output schema.

⚡ Agent Categories Agents fall into two categories: Chat Agents (invoked during user conversations) and Setup Agents (invoked once when "Market Analysis" is clicked). Setup agents do NOT grow context over conversation.

Chat Agents (invoked per user message)

🧭
Agent 1: CRO Router
Model: Gemini 3 Flash Preview | temp 0.2
Role: User-facing orchestrator. Classifies user intent, routes to sub-agents, merges outputs into conversational response.
Data: Receives full property JSON payload on first message. Remembers for session.
Output: { routing, proposals[], chat_response }
Orchestrator
📊
Agent 2: Property Analyst
Model: GPT-4o-mini | temp 0.1
Role: Analyses calendar gaps, restrictions (min stay), seasonal patterns, revenue forecasts.
Key Logic: Gap detection (1-night orphan, micro-gaps, last-minute vacancy), auto-revert intelligence.
Output: { gap_nights[], restrictions[], seasonal, revenue, summary }
Analysis
📈
Agent 3: Booking Intelligence
Model: GPT-4o-mini | temp 0.1
Role: Booking velocity, length of stay analysis, revenue breakdown, event correlation.
Key Logic: Velocity trend (accelerating/stable/decelerating based on occupancy %), LOS buckets.
Output: { velocity, length_of_stay, revenue, day_of_week, summary }
Analysis
🌍
Agent 4: Market Research
Model: GPT-4o-mini | temp 0.1
Role: Parses pre-cached market events, competitor rates, positioning verdict.
Key Logic: Event price factors (high=1.3x, medium=1.15x), positioning vs P50.
Output: { events[], competitors, positioning, summary }
Market Intel
🛡️
Agent 5: PriceGuard
Model: GPT-4o-mini | temp 0.0
Role: Pricing engine + final safety validator. Computes prices, clamps to floor/ceiling, validates, assigns risk.
Key Logic: 4-step pipeline: Compute → Clamp → Validate (6 rules) → Risk Level. Has unconditional veto power.
Output: { results[], batch_summary }
Validation
💬
Guest Reply Agent
Model: GPT-4o-mini | temp 0.3
Role: Drafts professional guest replies based on conversation history and booking context.
Output: Conversational reply text
Communication

04 Database Schema — 10 Tables

PostgreSQL (Neon) with Drizzle ORM. Schema defined in src/lib/db/schema.ts.

1. listings
Property registry — one row per Hostaway listing.
Column Type Description
id serial PK Internal ID
hostaway_id text UNIQUE Hostaway listing ID
name text Property name (e.g. "City Walk 1BR")
city, area text Location (e.g. "Dubai", "JBR")
bedrooms_number, bathrooms_number int Property specs
price numeric(10,2) Current nightly rate
currency_code varchar(3) "AED" by default
price_floor numeric(10,2) ⬇️ Minimum allowed price (guardrail)
price_ceiling numeric(10,2) ⬆️ Maximum allowed price (guardrail)
amenities jsonb String array of amenities
2. inventory_master
Daily price calendar — one row per listing per date. Source of occupancy calculations.
Column Type Description
id serial PK
listing_id int FK → listings
date date Calendar date
status text "available" | "reserved" | "booked" | "blocked"
current_price numeric(10,2) Active nightly rate for this date
min_stay, max_stay int Length of stay restrictions
proposed_price numeric(10,2) AI-proposed price (pending approval)
change_pct int % change from current
proposal_status text "pending" | "approved" | "rejected"
reasoning text AI reasoning for the proposal
📊 Occupancy Query SELECT COUNT(*), COUNT(CASE WHEN status IN ('reserved','booked') THEN 1 END) FROM inventory_master WHERE listing_id = ? AND date BETWEEN ? AND ?
3. reservations
Guest bookings from Hostaway. Used for booking intelligence and revenue analysis.
Column Type Description
id serial PK
listing_id int FK → listings
guest_name, guest_email text Guest details
start_date, end_date date Stay dates
channel_name text "Airbnb" | "Booking.com" | "Direct"
total_price, price_per_night numeric Financials
reservation_status text "confirmed" | "pending" | "cancelled"
4. market_events
AI-discovered events, holidays, and market signals from the Marketing Agent (internet search).
Column Type Description
id serial PK
listing_id int FK → listings null = portfolio-level
title text "Art Dubai" | "Ramadan"
start_date, end_date date Event dates
event_type text "event" | "holiday" | "competitor_intel" | "positioning"
expected_impact text "high" | "medium" | "low"
suggested_premium numeric(5,2) AI-suggested % price increase
confidence int 0–100 confidence score
description text Event description
5. benchmark_data
Competitor pricing intelligence from the Benchmark Agent. One row per listing + date range.
Column Type Description
p25_rate, p50_rate, p75_rate, p90_rate numeric Market rate distribution
avg_weekday, avg_weekend numeric Average competitor rates
recommended_weekday/weekend/event numeric AI-recommended target rates
verdict text "UNDERPRICED" | "FAIR" | "SLIGHTLY_ABOVE" | "OVERPRICED"
percentile int Your position in the market (e.g. 65th = FAIR)
comps jsonb[] Array of competitor listings (name, source, avgRate, rating)
6–10. Supporting Tables
Additional tables for conversations, settings, and guest management.
Table Purpose Key Columns
chat_messages AI conversation memory session_id, role, content, structured (jsonb)
user_settings User preferences & API keys user_id, lyzr_api_key, hostaway_api_key, preferences
guest_summaries AI summary of guest conversations listing_id, sentiment, themes[], action_items[]
hostaway_conversations Cached guest messages from Hostaway hostaway_conversation_id, messages (jsonb)
mock_hostaway_replies Shadow table for safe admin replies conversation_id, text

05 API Routes — Complete Reference

All Next.js API routes in src/app/api/.

Core Chat & Analysis

Method Route Description
POST /api/chat Main chat endpoint. Fetches data on first message, injects into Lyzr prompt, returns agent response.
POST /api/market-setup Triggers Market Analysis. Calls Marketing + Benchmark agents, saves events & benchmark to DB.
GET /api/events?listingId=X&dateFrom=Y&dateTo=Z Returns market events for sidebar display.
GET /api/benchmark?listingId=X&dateFrom=Y&dateTo=Z Returns benchmark data (P25/P50/P75) for sidebar.
GET /api/calendar-metrics?listingId=X&from=Y&to=Z Returns occupancy %, booked/available/blocked nights.

Property & Calendar Management

Method Route Description
GET /api/calendar?listingId=X&from=Y&to=Z Returns daily inventory (prices, status, min stay).
POST /api/calendar/block Blocks specific dates.
POST /api/calendar/unblock Unblocks specific dates.
PATCH /api/listings/:id Updates listing (floor price, ceiling price, etc).
GET /api/reservations?listingId=X Returns reservations for a listing.

Proposals

Method Route Description
POST /api/proposals/bulk-save Saves AI proposals to inventory_master.
POST /api/proposals/:id/approve Approves a single proposal.
POST /api/proposals/bulk-approve Approves all pending proposals for a listing.
POST /api/proposals/bulk-reject Rejects all pending proposals for a listing.

Data Sync (Hostaway → PostgreSQL)

Method Route Description
POST /api/sync Triggers full sync via Python backend (listings, calendar, reservations).
POST /api/sync/trigger Triggers sync for a specific data type.
GET /api/sync/status Returns sync status (last synced timestamps).

06 Step-by-Step Data Flows

Exactly what happens when each button is clicked and each query is made.

Flow 1: User clicks "Market Analysis" button

Frontend (unified-chat-interface.tsx) │ ├─ 1. Validate: Date range selected? ✓ Guardrails set? ✓ │ ├─ 2. POST /api/market-setup │ Body: { dateRange, context: { propertyId, propertyName }, userId } │ Backend (market-setup/route.ts) │ ├─ 3. SELECT listing from listings WHERE id = listingId │ ├─ 4. SELECT calendar metrics from inventory_master │ → occupancy_pct, booked_nights, avg_price │ ├─ 5. SELECT reservations from reservations │ → recent bookings for context │ ├─ 6. Build prompt for Marketing Agent (Sonar) │ "Search for events, holidays in {area} between {dateFrom}-{dateTo}" │ → Returns: events with impact, premium %, confidence │ ├─ 7. UPSERT events into market_events table │ (unique on listing_id + title + start_date) │ ├─ 8. Build prompt for Benchmark Agent (Sonar) │ "Find competitor rates for {bedrooms}BR in {area}" │ → Returns: P25/P50/P75/P90, recommended rates, competitor list │ ├─ 9. UPSERT into benchmark_data table │ └─ 10. Return { eventsCount, duration, sqlTrace[] } Frontend receives response: ├─ Toast: "Market Analysis Complete — X events in Y seconds" ├─ Sidebar refreshes (events + benchmark widgets) └─ Chat area stays clean — ready for user's first message

Flow 2: User sends first message (e.g. "Hi" or "Give me the analysis")

Frontend (unified-chat-interface.tsx) │ ├─ 1. POST /api/chat │ Body: { message, context: { type, propertyId, propertyName }, dateRange, sessionId } │ Backend (chat/route.ts) │ ├─ 2. Check: Is this the first NON-SYSTEM user message in this session? │ SELECT id FROM chat_messages │ WHERE session_id = ? AND role = 'user' AND content NOT LIKE '[SYSTEM]%' │ → needsDataInjection = (count == 0) │ ├─ 3. IF needsDataInjection: │ ├─ SELECT * FROM listings WHERE id = listingId │ │ → property: { name, area, current_price, floor_price, ceiling_price } │ │ │ ├─ SELECT COUNT(*), occupancy FROM inventory_master │ │ WHERE listing_id = ? AND date BETWEEN ? AND ? │ │ → metrics: { occupancy_pct, booked_nights, bookable_nights } │ │ │ ├─ SELECT * FROM reservations │ │ WHERE listing_id = ? AND dates overlap range │ │ → recent_reservations: [{ guestName, dates, totalPrice, channel }] │ │ │ ├─ SELECT * FROM benchmark_data │ │ WHERE listing_id = ? AND date range matches │ │ → benchmark: { p25, p50, p75, recommended_weekday/weekend/event } │ │ │ ├─ SELECT * FROM market_events │ │ WHERE listing_id = ? AND dates overlap range │ │ → market_events: [{ title, dates, impact, premium }] │ │ │ └─ Build propertyDataPayload (all numbers, no strings) │ ├─ 4. SAVE user message to chat_messages │ ├─ 5. Construct anchoredMessage: │ "[SYSTEM: CURRENT PROPERTY DATA] │ {JSON.stringify(propertyDataPayload)} │ [/SYSTEM] │ User Message: {message}" │ ├─ 6. POST to Lyzr Agent Studio │ { agent_id, session_id, message: anchoredMessage } │ ├─ 7. Parse Lyzr response → extract JSON { routing, proposals, chat_response } │ ├─ 8. SAVE agent reply to chat_messages │ └─ 9. Return response to frontend Frontend receives response: ├─ Display chat_response as markdown ├─ If proposals[] is not empty → show proposal cards └─ Proposal cards have "Deploy to Control" / "Discard" buttons

Flow 3: User edits Price Guardrails

PriceGuardrailsEditor component │ ├─ 1. User clicks guardrails badge → inline editor opens ├─ 2. User enters floor (e.g. 600) and ceiling (e.g. 1800) ├─ 3. Validation: floor > 0, ceiling > floor │ ├─ 4. PATCH /api/listings/:id │ Body: { priceFloor: 600, priceCeiling: 1800 } │ ├─ 5. UPDATE listings SET price_floor = 600, price_ceiling = 1800 │ └─ 6. UI updates → Market Analysis button becomes enabled

07 Price Guardrails & Safety System

How PriceOS prevents bad pricing decisions.

PriceGuard 4-Step Pipeline

🔢
Step 1: Compute
base × event_factor × occupancy_factor
Uses benchmark recommended rates as base.
Calculation
📏
Step 2: Clamp
if price < floor → price = floor
if price > ceiling → price = ceiling
No exceptions.
Hard Limit
✅
Step 3: Validate
6 rules checked in order. Stop at first failure:
• Floor check (REJECT)
• Ceiling check (REJECT)
• ±50% swing (REJECT)
• P25/P75 market outlier (FLAG)
Validation
⚡
Step 4: Risk Level
<5% change → Low (auto-approve OK)
5-15% → Medium (human approval)
>15% → High (human approval)
Any event → at least Medium
Classification

Date Classification System

Classification Criteria Action
🟢 Protected High-impact event OR occupancy > 70% Do NOT discount. Premium pricing.
🔵 Healthy > 45 days out, priced appropriately Monitor only. No action needed.
🟡 At Risk < 30 days out, below target occupancy Consider price adjustment.
🔴 Distressed < 14 days out, vacant, no demand Aggressive LOS relaxation + discount.

Verdict System

Verdict Meaning Can Deploy?
APPROVED All rules passed. Safe to deploy. ✅ Yes
FLAGGED Market outlier or large change. Review recommended. ⚠️ With caution
REJECTED Violates floor/ceiling or ±50% limit. Cannot deploy. ❌ No
