PriceOS Agent Orchestration

A 7-agent AI system for Dubai short-term rental dynamic pricing. Every price proposal is grounded in real market data — verified, guardrailed, and fully auditable.

System Overview
Two-Phase Architecture
PriceOS operates in two distinct phases: Setup (internet research) and Chat (database-driven analysis). Data flows one-way — internet → database → agents → user.
🔍
Phase 1 — Market Analysis (Setup)
Triggered once by user

User selects a date range and property, then clicks Market Analysis. Two internet-search agents run in parallel (Marketing Agent + Benchmark Agent), scraping Airbnb, Booking.com, and event sites. Results are saved to the database and consolidated into a Lyzr Global Context.

💬
Phase 2 — Pricing Chat
Continuous conversation

Chat is unlocked after Setup. The CRO Router receives user messages and reads the Lyzr Global Context to generate pricing proposals. All 5 chat agents operate exclusively from the pre-loaded Global Context — zero database or internet access during chat. PriceGuard validates every proposal before delivery.

Agent Architecture
7 Specialised Agents
Each agent has a strict, single responsibility. No agent is created or modified from application code — all are managed in Lyzr Studio using their permanent Agent IDs.
⚡ Setup Agents — Internet Research (runs once per Market Analysis click)
Agent 6 · Marketing_Agent_ID
Marketing Agent
Searches the web for Dubai events, public holidays, demand trends, competitor pricing, and weather outlook. Returns structured JSON saved to market_events table.
🌐 Internet Setup Only Writes market_events
Agent 7 · LYZR_Competitor_Benchmark_Agent_ID
Benchmark Agent
Scrapes Airbnb, Booking.com, Vrbo for 10–15 comparable properties. Calculates P25/P50/P75/P90 rate distribution and weekday/weekend splits. Writes to benchmark_data table.
🌐 Internet Setup Only Writes benchmark_data
💬 Chat Agents — Global Context Readers (active during conversation)
Agent 1 · AGENT_ID (CRO Router)
CRO Router
User-facing conversational agent. Classifies intent, routes to sub-agents in parallel, merges outputs, and delivers a formatted pricing response. No database access — orchestration only.
📋 Global Context Chat Phase
Agent 2 · Property Analyst
Property Analyst
Reads the inventory array and external_signals from the Global Context. Returns gap nights, restriction flags, seasonal averages, and revenue forecasts. Zero database access.
📋 Global Context Chat Phase
Agent 3 · Booking Intelligence
Booking Intelligence
Infers booking patterns from consecutive booked days in the inventory array. Returns velocity, length-of-stay, revenue, event correlation, and benchmark comparison from external_signals. Zero database access.
📋 Global Context Chat Phase
Agent 4 · Market Research
Market Research
Parses external_signals.events_analysis and external_signals.market_analysis from the Global Context. Returns event price factors, competitor P50/P75/P90 comps, positioning verdict, and recommended rates. Zero DB or internet access.
📋 Global Context Chat Phase
Agent 5 · PriceGuard
PriceGuard
Called last, always. Reads property.price_floor, property.price_ceiling, and market rate bands (P25–P90) from external_signals in the Global Context. Returns APPROVED / REJECTED / FLAGGED. Zero database access.
📋 Global Context Chat Phase
Market Analysis Flow
What happens when user clicks "Market Analysis"
A complete end-to-end sequence from button click to the Pricing Analyst being grounded in real data.
1
User Clicks "Market Analysis"
Frontend sends POST /api/market-setup with listing_id + dateRange.
2
Next.js Proxies to Python Backend
Validated and forwarded to POST /api/agent/market-analysis. User API key is fetched from user_settings table.
3
Internet Research (Parallel)
Two Lyzr agents are triggered simultaneously with a minimal static message — no dynamic prompts injected.
agent.run("Run your market event analysis.") # Marketing Agent agent.run("Run your competitor benchmark analysis.") # Benchmark Agent
4
Save Results to Database
Agent responses are parsed and written to Neon PostgreSQL. Old records for the date range are deleted first (physical replacement pattern).
DELETE FROM market_events WHERE listing_id=? AND date IN range INSERT INTO market_events (title, event_type, start_date, ...) DELETE FROM benchmark_data WHERE listing_id=? AND date_from=? INSERT INTO benchmark_data (p50_rate, comps, verdict, ...)
5
Fetch Internal Property Data
Query the database for the selected property details and calendar inventory.
SELECT * FROM listings WHERE id = listing_id SELECT * FROM inventory_master WHERE listing_id = ? AND date BETWEEN start AND end
6
Consolidate → Push Global Context
All data is merged into a single JSON object and pushed to the Pricing Analyst agent as a Lyzr Global Context named active_property_data. Old context is deleted first.
✓
Chat Unlocked
Frontend receives success. The chat interface is now active and the Pricing Analyst is fully grounded in the latest real-world data.

Global Context Pattern
Physical Replacement — One Context, Always Fresh
The Lyzr Global Context named active_property_data is the only data channel. It is replaced atomically on every Market Analysis run.
The backend runs a strict 3-step context update cycle. This guarantees the Pricing Analyst never uses stale data. Context name is always fixed — no unique names per run, no accumulation.
✅Step 1 — Delete: List all Lyzr contexts. Find any named active_property_data. Delete them all.
✅Step 2 — Create: Create a fresh context named active_property_data with the full consolidated JSON.
✅Step 3 — Attach: Call agent.add_context(ctx) to attach to the Pricing Analyst Studio agent.
Python — update_context()
# 1. CLEANUP — delete old context all_contexts = studio.contexts.list() for ctx in all_contexts: if ctx.name == "active_property_data": studio.contexts.delete(ctx.id) # 2. CREATE — fresh context with new data ctx = studio.create_context( name="active_property_data", value=json.dumps(consolidated_data) ) # 3. ATTACH — bind to Pricing Analyst agent.add_context(ctx)
Engineering Guardrails
Immutable Rules — Enforced in Code
These constraints are structurally enforced — not enforced by prompts. Violations raise exceptions at startup, not at runtime.
✅ What the code DOES
✅Get agents by ID only: studio.get_agent(agent_id=...)
✅Update Global Context only: studio.create_context(...) + agent.add_context(...)
✅Use static trigger messages: agent.run("Run your analysis.")
✅Raise ValueError on missing IDs: fails fast at startup, not silently at runtime
✅Read API keys from environment: no hardcoded keys anywhere in source code
✅Physical replacement: delete old context before creating new one
🚫 What the code NEVER does
🚫No agent creation: studio.create_agent(...) — removed from all files
🚫No prompt injection: no f-string prompts with dates, listing IDs, or property names
🚫No instruction updates: agent system prompts are never set from code
🚫No hardcoded API keys: all keys from .env → settings.*
🚫No context accumulation: old context is always deleted before adding new one
🚫No direct DB access from agents: Pricing Analyst reads ONLY from Global Context
